# napl-core (Rust)

Phase 1 of an oxc-style Rust rewrite of the NAPL toolchain: a faithful port of
the **pure** core algorithms from the TypeScript `@napl/core` and the LSP
scanner. No I/O, no async, no agent runner, no CLI, no LSP server yet — those are
later phases.

## Layout

```
rust/
  Cargo.toml                 workspace (resolver 2, shared clippy-pedantic lints)
  rust-toolchain.toml        stable channel + rustfmt/clippy
  fixtures/                  cross-check fixtures produced BY the TS build
  crates/napl-core/
    src/
      text_diff.rs           LCS unified diff + hunk parser (byte-identical to TS)
      blame.rs               patch-replay line ancestry
      hash.rs                sha256 hex content hashing
      body_lines.rs          frontmatter/body line-offset math
      scanner.rs             prompt span scanner (UTF-16 positions)
      extensions.rs          prompt/machine extension alias logic
      schemas/               serde types + validators
        line_range.rs        [n] / n -> [n,n] normalization
        attribution.rs  ml.rs  ir.rs  map.rs  journal.rs  lock.rs  frontmatter.rs
        ordered_map.rs       insertion-order-preserving string map
    tests/cross_check.rs     fixture-driven parity tests
```

## Verification

```
cargo build
cargo test          # 122 unit tests + 5 cross-check tests
cargo clippy --all-targets -- -D warnings
cargo fmt --check
```

## Cross-check fixtures

`fixtures/*.json` are **generated by running the built TypeScript implementation**
(`packages/core/dist`, `packages/lsp/dist`) — the expected values are produced by
TS itself, not hand-written. Regenerate with the scratchpad generator that calls
`unifiedDiff`, `contentHash`, `blameFile`, `scanDocument`, and the zod validators
via `node -e` against `dist`. `tests/cross_check.rs` reads them and asserts the
Rust port reproduces:

- `diff_cases.json` — `unifiedDiff` output byte-for-byte (plus a parse+apply roundtrip).
- `hash_cases.json` — `contentHash` hex.
- `blame_cases.json` — `blameFile` per-line generation vectors.
- `scanner_cases.json` — `scanDocument` module/deps/refs/region spans, including
  the astral-emoji (`😀😀@ref`) and BMP-multibyte (`é@ref`) UTF-16 column cases.
- `schema_cases.json` — accept/reject decisions for attribution, ml, ir, lock, map.

## Parity notes

**Matches TS exactly**

- `text_diff`: same greedy LCS + tie-break (`dp[i+1][j] >= dp[i][j+1]` → delete),
  same `context = 3` hunk grouping, same `@@ -a,b +c,d @@` headers, same
  `toLines` trailing-newline / empty-input behavior. Output is byte-identical
  (hand-rolled LCS — the `similar` crate is *not* used, so nothing can drift the
  format).
- `blame`: identical patch-replay (added / modified / moved-by-insertion /
  created-then-edited / untouched-keeps-oldest-gen), same fallback-gen rule.
- `hash`: sha256 → lowercase hex, verified against a known vector and TS output.
- `body_lines`: identical body-start detection and 1-based mapping.
- `scanner`: positions are **zero-based `line` / `character` in UTF-16 code
  units** (documented on `Position`), matching the LSP model and TS `.length` /
  regex `.index` semantics; verified against TS spans including multibyte content.
- schemas: acceptance set matches the zod schemas (line-range `[n]`/`n` → `[n,n]`,
  `no-op` kind, ZWJ + 1-2-codepoint alias rules, map v2 `unattributed` marker,
  journal corrupt-line skipping, IR contract union). Error *messages* differ (as
  allowed); accept/reject decisions are identical and cross-checked.
- `extensions`: codepoint-correct suffix matching; the curated emoji list and the
  `machineExtensionForPrompt` mirror rule are reproduced exactly.

**Intentional deviations (scope, not blockers)**

- File-system entry points (`readMap`/`writeMap`, `readJournal`/`appendJournalEntry`
  from disk, `findPromptFiles`) are out of phase-1 scope. Their pure cores *are*
  ported: `parse_map` / `map_to_json`, `read_journal_str` (string in, warnings
  out), and `is_prompt_file` / `prompt_extensions`.
- Validation error strings are new Rust messages; only accept/reject parity is
  guaranteed (and enforced by the fixtures).
- `map_to_json` emits `serde_json` pretty output (2-space, trailing newline) that
  round-trips through `parse_map`; byte-identity with TS `JSON.stringify` is not
  claimed (only `text_diff` output is contracted to be byte-identical).

**Port-blockers discovered:** none. Every pure algorithm reproduced exactly.

**Maintenance caveat:** `serde_yaml` (0.9) is used for frontmatter and is
upstream-deprecated; it is the only mainstream serde YAML option and compiles
cleanly. A future phase may swap it for a maintained YAML parser.
